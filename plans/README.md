## Dependency notes

- Plan 001 is a master roadmap. Its Phase 0 contract probe must finish before
  production Hostify mutations or checkout work begins.
- Split Phases 1-7 into separate implementation plans after Phase 0 records the
  account-specific behavior and the project selects its PostgreSQL access
  library, migration tool, Redis service, and worker runtime.
- Plan 002 is the best next backend task now that listing cache, rate limiting,
  and observability exist. It turns cached listings into stable public backend
  contracts for the frontend.
- Plan 003 should follow Plan 002 because live quotes need to verify the
  listing exists in the local catalog and must return a frontend-safe product
  contract.
- Plan 004 should follow Plan 003 because draft orders must be created from a
  retrievable quote snapshot, not from stale catalog data or client-submitted
  totals.

## Findings considered and rejected

  rejected because several endpoints are destructive, account-dependent,
  sensitive, deprecated, or explicitly unavailable.
- Keep MongoDB as a mandatory Hostify catalog store from day one: rejected for

  sanitized `jsonb` can hold the documented listing projection with less
  operational complexity. Reconsider only with measured evidence.
- Replace Stripe and Hostkit with Hostify payment/accounting/check-in endpoints:

  payment authority, Portuguese fiscal-document creation, credit-note creation,
